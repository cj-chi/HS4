# 臉部幾何一致對齊：大規模研究計畫

**目標**：讓人物卡與原始 JPG 在正面投影下具備**幾何一致的臉部特徵**（特別是臉邊緣輪廓），達成「同一張圖 → MediaPipe landmark → 產卡 → 遊戲截圖 → MediaPipe → 兩次 landmark 的幾何量一致」。

**指導綱領**：[臉邊緣_Slider_幾何一致清查.md](臉邊緣_Slider_幾何一致清查.md)  
**方法基礎**：[還原與FACE_OVAL某段一致的量_方法.md](還原與FACE_OVAL某段一致的量_方法.md)、[ShapeAnime_還原FaceBase_2D投影量_計算步驟.md](ShapeAnime_還原FaceBase_2D投影量_計算步驟.md)

---

## 一、研究範疇與核心問題

### 1.1 核心問題

依據清查，臉邊緣相關的 **13 個 2D 可見 Slider（index 0, 2, 4, 5, 6, 8, 9, 10, 11, 13, 15, 16, 18）** 目前全部「做不到」幾何一致，因為：

- **Source 端（遊戲）**：沒有精確的「x = 某幾何量」定義；必須從 **cf_customhead + ShapeAnime + 頭部 mesh/骨架/蒙皮** 還原「Slider → 2D 投影輪廓 → x」。
- **Landmark 端（MediaPipe）**：有 FACE_OVAL 36 點與其他特徵點，可算任何幾何量，但若與遊戲端的 x **定義不同**，則無法對齊。

### 1.2 解決方案概念

1. **還原 Source 端的 x**：從遊戲資源（abdata）取出頭部 mesh、骨架、蒙皮、ShapeAnime、cf_customhead，建立「Slider → 骨骼變換 → 2D 投影輪廓 → x」管線。
2. **在 Landmark 端實作同一 x**：對還原出的每個 x，在 FACE_OVAL 與 MediaPipe 468 點上用**完全相同的定義**（同一組點、同一量法、同一正規化）計算。
3. **建立雙向映射**：rate/slider → x、x → rate/slider，供 landmark → 產卡使用。
4. **全面驗證**：對每個 Slider，用「原圖 → 產卡 → 遊戲截圖 → landmark」閉環驗證幾何一致性。

### 1.3 研究範疇

| 範疇 | 包含 | 不包含 |
|------|------|--------|
| **Slider 範圍** | 臉邊緣 2D 可見的 13 個（0, 2, 4, 5, 6, 8, 9, 10, 11, 13, 15, 16, 18） | 深度相關（1, 3, 7, 12, 14, 17）、五官細部（眼、鼻、口內部） |
| **幾何特徵** | 臉緣輪廓（FACE_OVAL）、多段高度的臉寬、下巴/顴/頰段寬度與形狀 | 膚色、紋理、表情 BlendShape |
| **驗證條件** | 正面投影、中心與比例對齊後的幾何量一致（比例對比例、距離對距離） | 側面、斜視角、3D 深度 |

### 階段 2：臉型核心 - index 4, 5（5～7 天）

**目標**：對 **index 4（head_lower_width）** 與 **index 5（jaw_width）** 重複階段 1 流程。

### 階段 3：下巴形狀 - index 6, 11（5～7 天）

**目標**：處理涉及「高度」的 **index 6（ChinY）** 與 **index 11（ChinTipY）**。

### 階段 4：頰與臉上部 - index 2, 13, 15, 16, 18（10～12 天）

**目標**：處理額頭、顴、頰（5 個 Slider，定義可能較複雜）。

### 階段 5：其他 2D 可見 - index 8, 9, 10（5～7 天）

**目標**：處理角度、高度、小範圍寬度（3 個 Slider）。

### 階段 6：整合與全面驗證（7～10 天）

**目標**：整合所有 13 個 Slider，進行大規模測試。

**整合產出**：
- `data/face_edge_sliders_geometry_map.json`
- `compute_face_edge_features.py`（計算所有 x）
- 更新 `ratio_to_slider_map.json` 與產卡流程

**驗證**：
- 單一 Slider 測試：65 張卡（每 Slider × 5 值），誤差 <5%
- 多 Slider 組合：10 組真實人臉，FACE_OVAL Hausdorff 距離 <10px
- 閉環測試：100 張隨機臉，平均誤差 <8%

## 三、關鍵技術挑戰與解決方案

### 3.1 遊戲端：從 ShapeAnime 還原 2D 投影量

**挑戰**：ShapeAnime 只給「rate → 骨骼 local」，需建立「骨骼 → 蒙皮 → 2D」管線。

**解決方案**：
1. **A. Unity 重現**（較慢但可行）：在 Unity 載入頭部、設 Slider、正交相機截圖或讀頂點 → 手動量或程式算輪廓 → x。
2. **B. 離線工具**（推薦）：Python/C# 自寫：解析 mesh、骨架、ShapeAnime → local → world → 蒙皮（`P_world = Σ weight_k × Bone_k_world × rest_pos_k`） → 投影 → 算 x。

### 3.2 Landmark 端：FACE_OVAL 輪廓插值

**關鍵**：在正規化高度 h 處，對 FACE_OVAL 36 點做線段插值求交點。

**實作**：
```python
# 對給定 h ∈ [0,1]
y_target = y_bottom + h * face_h
# 對每條線段 (P[i], P[i+1]) 查 y=y_target 的交點
x_left, x_right = compute_contour_width_at_height(FACE_OVAL, y_target)
w(h) = x_right - x_left
x = w(h) / face_h
```

### 3.3 定義一致性驗證

**方法**：對同一張已知 Slider 值的卡片截圖，兩端算出的 x 應一致。

**驗證流程**：
1. 在遊戲設 Slider=某值 → 截圖 → MediaPipe → x(landmark)
2. 用同一 Slider 值 → 遊戲端管線 → x(game)
3. 比對：|x(game) - x(landmark)| / x(game) < 5%
